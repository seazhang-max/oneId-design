# OneID 系统需求文档

## 一、阶段1：数据采集与标准化清洗

**核心目标**：确保进入图计算的数据质量，定义数据的"生杀大权"。

- **支持多源异构数据接入与清洗规则配置**：支持配置不同数据源（日志、DB、API）的清洗脚本，剔除脏数据。
- **支持全局黑白名单机制（Black/White List）**：
  - **黑名单过滤**：在入库前强制隔离特定 ID（如测试账号、爬虫设备、公共 Wi-Fi IP），禁止其生成边表数据。
  - **白名单兜底**：强制保留关键业务 ID，防止被误清洗。
- **支持 ID 类型优先级与置信度预定义**：在清洗阶段即标记 ID 的可信等级（如：实名认证 > 手机号 > 设备 IMEI > Cookie），为后续冲突消解做基础数据准备。

## 二、阶段2 & 3：图数据加载与 WCC 算法执行

**核心目标**：构建连通图，处理计算性能与异常结构。

- **支持异常大簇（超级节点）自动识别与熔断**：
  - **监控告警**：当某节点关联边数超过阈值（如 10万+），自动触发告警。
  - **自动处理**：支持配置策略，自动切断高频噪声节点的连接或暂停该簇计算，防止资源耗尽。
- **支持 ID 生命周期与衰减策略配置**：支持配置边的有效期（如：同设备+同IP 仅 24h 有效，Cookie 30天失效），在图加载时自动剪枝过期关系，保持图的"新鲜度"。
- **支持增量与全量计算的平滑切换**：支持日常增量实时 Mapping，以及定期（如每周）的全量重跑（Re-mapping），以修复增量计算积累的误差。

## 三、阶段4：ID 映射决策与冲突消解

**核心目标**：确定最终 OneID，处理复杂的人工干预与自动化决策。

- **支持可视化的关系边（Edge）权重动态调整**：允许在界面上调整不同 ID 组合（如手机号-设备号）的合并置信度分数，直接影响 WCC 后的连通结果。
- **支持 OneID 分裂（Split）与解绑机制**：针对错误合并（如家庭共用账号），支持手动或基于规则将特定 ID 从当前 OneID 中剥离，生成新 OneID。
- **支持人工干预合并与审批流（Workflow）**：
  - **操作**：支持强制合并两个独立的 OneID。
  - **管控**：对于批量或高风险操作，需支持"发起-审核-生效"流程。
  - **模拟（Dry Run）**：执行前预览影响范围（波及多少用户、下游哪些报表）。
- **支持查看 User 与当前 OneID 的关系拓扑图**：提供可视化界面，直观展示某个 OneID 下挂载的所有子 ID 及其关联来源，辅助判断合并是否正确。
- **支持跨层 OneID 合并指导与配置**：
  - **全局 OneID 指导下层合并**：全局层计算完成后，全局 OneID 可作为参考，指导集团层或品牌层在 WCC 冲突消解阶段的合并决策（例如：当集团/品牌层无法仅凭本层数据判定两个 ID 是否属于同一人时，可引用全局 OneID 的合并结论作为辅助依据）。
  - **按集团/品牌维度可配置**：支持针对每个集团或品牌独立配置是否引用上层 OneID（全局/集团）进行合并指导，配置粒度为：
    - **集团级配置**：控制该集团下的品牌是否引用集团 OneID 或全局 OneID 进行合并。
    - **品牌级配置**：控制该品牌是否引用集团 OneID 或全局 OneID 进行合并。
  - **配置项包括**：是否启用跨层合并指导、引用的目标层级（全局/集团）、冲突时的优先级策略（本层优先 vs 上层优先）。
  - **注意**：引入跨层合并指导后，集团/品牌层的 OneID 可能因上层 ID 的介入而被合并得更大（即原本在本层数据范围内不会连通的 ID，因全局/集团 OneID 的关联而被合并为同一个 OneID），这是预期行为，业务方需知晓并接受此合并粒度的变化。

## 四、阶段5：属性融合与画像沉淀

**核心目标**：丰富 OneID 的业务价值，确保画像准确。

- **支持属性合并与更新策略配置**：当多个 ID 汇聚为一个 OneID 时，支持配置属性覆盖优先级（如：转化数据源的性别 > 注册数据源的性别），自动填充用户画像宽表。
- **支持各层 OneID 与原始 ID 映射关系表生成**：支持按业务需求，将品牌层、集团层或全局层的 OneID 数据与该层范围内的原始业务 ID（如电商 UID、外卖 UID、APP UID 等）进行关联，生成对应层级的 OneID-原始ID 映射关系表，供下游业务系统直接查询使用。映射表生成需支持：
  - **按层级生成**：可分别生成品牌层、集团层、全局层的映射关系表，各层映射表独立维护。
  - **按业务线筛选**：支持按业务线（如电商、外卖、APP 等）筛选原始 ID 类型，生成特定业务线的映射子集。
  - **映射关系包含**：OneID、原始 ID 类型、原始 ID 值、关联来源（哪个业务系统/数据源）、置信度分数、生效时间等字段。

## 五、阶段6 & 7：跨层映射表、服务发布与审计

**核心目标**：保障数据安全，提供"后悔药"与追溯能力。

- **支持全链路变更历史审计（Audit Log）**：详细记录每一次 OneID 变更（自动合并、人工拆分、属性更新）的时间、原因、操作人及变更前后的快照。
- **支持多维度版本回滚机制**：
  - **单用户回滚**：针对特定 User/OneID 回滚到指定历史版本。
  - **全量时间点回滚**：支持将整个 OneID 映射表回滚到某天（如昨天）的状态，用于应对重大算法事故。
  - **任务级回滚**：支持撤销某次特定的人工干预任务。
- **支持历史数据回溯（Data Backfill）**：当 OneID 发生变更后，支持触发下游数仓或画像表的批量修正任务，确保历史报表数据的一致性。
- **支持多业务域（Domain）隔离视图**：在服务发布层，支持按业务线（如电商域、内容域）查看和导出独立的 OneID 映射服务，互不干扰。
